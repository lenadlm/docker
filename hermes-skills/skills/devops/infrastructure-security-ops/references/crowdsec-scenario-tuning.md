# CrowdSec Scenario Tuning & Metrics Interpretation

## Scenario Lifecycle

```
Parser matches log line → sets Meta fields
  ↓
Scenario filter evaluates → matches? → bucket created/updated for this IP
  ↓
Bucket fills to capacity within leakspeed → OVERFLOW → alert generated
  ↓
Alert creates ban decision → LAPI → bouncer enforces at CROWDSEC_CHAIN
```

## Scenario Anatomy (iptables-scan-multi_ports)

```yaml
type: leaky                              # Leaky bucket = drains over time
name: crowdsecurity/iptables-scan-multi_ports
filter: "evt.Meta.log_type == 'iptables_drop' && evt.Meta.service == 'tcp'"
groupby: evt.Meta.source_ip              # One bucket per IP
distinct: evt.Parsed.dst_port            # Only count UNIQUE ports
capacity: 5                              # Fill bucket = need this many hits
leakspeed: 40s                           # Bucket drains this fast
blackhole: 1m                            # Cooldown after overflow
```

**Key insight:** `distinct: dst_port` means hitting the same port repeatedly doesn't advance the bucket. The IP must hit 5 *different* ports within 40s for a ban.

## Threshold Trade-offs

| Capacity | Leakspeed | Rate | Catches | False-pos Risk |
|----------|-----------|------|---------|----------------|
| 15 | 5s | 3.0 ports/sec | Ultra-fast scanners only | Very low |
| 10 | 30s | 0.33 ports/sec | Moderate scans | Low |
| 5 | 40s | 0.125 ports/sec | Most real scanners | Medium |
| **3** | **10s** | **0.3 ports/sec** | **Current — catches most scans fast** | **Low-Medium** |
| 2 | 10s | 0.2 ports/sec | Near-instant bans | Higher risk |

**Rule of thumb:** `capacity (distinct ports) / leakspeed (seconds)` = the scanning rate that triggers a ban. Lower values = easier to trigger.

## Understanding `cscli metrics` for Scenarios

```
| Scenario                                | Current Count | Overflows | Instantiated | Poured | Expired |
| crowdsecurity/iptables-scan-multi_ports | 6             | -         | 455          | 799    | 447     |
```

| Column | Meaning | What to watch |
|--------|---------|---------------|
| **Current Count** | Active buckets right now (IPs being tracked) | >0 means scanning is detected. Each value = 1 IP actively hitting different ports |
| **Overflows** | Total bans generated by this scenario ever | **If `-` (zero), no bans ever fired.** Scenario is tracking but never triggered. This is why WB console shows nothing |
| **Instantiated** | Total buckets ever created (all-time) | Growing = scenario is alive and receiving events |
| **Poured** | Total events that match the scenario filter and were poured into buckets | Growing = filter match is working. This is the input volume to the leaky bucket |
| **Expired** | Buckets that drained before overflowing | High expired + zero overflows = threshold is too high or window too short |

**Critical:** An overflow IS a ban decision. If Overflows is `-`, the scenario has never generated a ban — regardless of Current Count, Poured, or Instantiated numbers. Active buckets alone do not appear in the Web Console.

## Why the Web Console Shows Nothing

The CrowdSec Web Console (app.crowdsec.net) shows:
- **Active decisions/bans** (not active scenarios)
- Alerts pushed to the Central API

A scenario with **0 overflows** has generated **0 decisions** → **0 alerts** → **nothing in the console.**

To check locally:
```bash
# See if any ban decisions exist from this scenario
sudo cscli alerts list --scenario crowdsecurity/iptables-scan-multi_ports

# See all active firewall bans
sudo cscli decisions list

# Check scenario status (not just decisions)
sudo cscli scenarios list | grep iptables-scan
```

## Scenario State: `enabled,local`

When a scenario is edited directly in `/etc/crowdsec/scenarios/` (instead of via `cscli scenarios install`):

```
crowdsecurity/iptables-scan-multi_ports  🏠  enabled,local  /etc/crowdsec/scenarios/...
```

- `🏠 enabled,local` = installed but locally modified
- Won't be auto-upgraded by `cscli hub upgrade`
- Still fully functional — the local file overrides the hub version
- To revert to hub default: `sudo cscli scenarios upgrade crowdsecurity/iptables-scan-multi_ports`

## Verification Workflow After Tuning

```bash
# 1. Apply changes
sed -i 's/capacity: 10/capacity: 5/' /etc/crowdsec/scenarios/iptables-scan-multi_ports.yaml
sed -i 's/leakspeed: 30s/leakspeed: 40s/' /etc/crowdsec/scenarios/iptables-scan-multi_ports.yaml

# 2. Reload CrowdSec
sudo systemctl reload crowdsec

# 3. Wait 60+ seconds for new events to accumulate
sleep 60

# 4. Check metrics — look for Overflows > 0
sudo cscli metrics | grep iptables-scan

# 5. Check for actual decisions
sudo cscli alerts list --scenario crowdsecurity/iptables-scan-multi_ports

# 6. Verify parsing pipeline
sudo cscli metrics | grep -E "(kern\.log|iptables-logs)"
```

## Testing via Log Injection

To verify the full pipeline without waiting for a real scan:

```bash
TEST_IP="10.0.0.99"
NOW=$(date '+%Y-%m-%dT%H:%M:%S+03:00')
for port in 22 80 443 3306 8080; do
  echo "$NOW srv kernel: [UFW BLOCK] IN=eth0 OUT= MAC=... SRC=$TEST_IP DST=192.168.1.100 LEN=60 TOS=0x00 PREC=0x00 TTL=64 ID=12345 DF PROTO=TCP SPT=40000 DPT=$port WINDOW=64240 RES=0x00 SYN URGP=0" | sudo tee -a /var/log/kern.log > /dev/null
done
sleep 5
sudo cscli alerts list --scenario crowdsecurity/iptables-scan-multi_ports
```

**Caveats:**
- Use `sudo sh -c '...'` heredoc-style to avoid shell expansion artifacts
- Match exact real UFW BLOCK format (no `DF` flag — ParseKV chokes on non-key=value tokens in the middle of KV pairs)
- Each CrowdSec reload resets in-memory buckets. Inject AFTER the final reload.
- CrowdSec tracks log position via state files — if the log was already consumed past the injection point, the entries won't be read. Use fresh timestamps matching the current time.

### Diagnosis when test injection doesn't produce a ban

If you inject 5 UFW BLOCK entries (5 distinct ports, same IP, fresh timestamps) but get 0 overflows:

1. **Check parser hits:** `sudo cscli metrics | grep iptables-logs` — should show `Hits > 0`. If 0, the parser filter isn't matching the log format.
2. **Check format match:** Run `sudo tail -1 /var/log/kern.log | grep -c "UFW BLOCK"` to verify entries are in the log. Then compare the exact format to a real UFW BLOCK line — the `DF` flag is the most common mismatch.
3. **Check `[UFW AUDIT]` exclusion:** The parser explicitly filters out `ACCEPT` and `[UFW AUDIT]`. Everything with `IN=` that is NOT ACCEPT/AUDIT passes. `[UFW BLOCK]` should match.
4. **Check ParseKV behavior:** The parser uses `ParseKV(evt.Parsed.message, ...)` — this parses `KEY=VALUE` pairs. Tokens like `DF` (no `=`) break sequential key-value parsing. Real UFW BLOCK lines don't have `DF` but sometimes have `URGP=0` as the last field (which is fine since it's after all KV pairs).
5. **Check acquis config:** Ensure kern.log has `type: syslog` in the acquis config, NOT `type: syslog+iptables-logs`. The parsers are chained by stage (s00-raw/s01-parse), not by acquis label.
6. **Check fail2ban filtering conflicting:** Ensure the fail2ban portscan jail (if still present) isn't consuming/re-positioning the log. fail2ban and CrowdSec can both read kern.log without conflict.
7. **Check for `enabled,local` state:** `sudo cscli scenarios list | grep iptables-scan`. The `🏠 local` tag means the file was edited outside hub management — still functional but a potential source of confusion.

## Parser Pipeline Notes

```
kern.log line:
  2026-06-21T20:55:00+03:00 srv kernel: [UFW BLOCK] IN=eth0 OUT= SRC=1.2.3.4 ...

syslog-logs parser (s00-raw):
  → Extracts: timestamp, program='kernel', message='[UFW BLOCK] IN=eth0 ...'

iptables-logs parser (s01-parse):
  → Filter: program == 'kernel' AND message contains 'IN='
  → Excludes: ACCEPT, AUDIT
  → Sets: log_type='iptables_drop', service='tcp', source_ip='1.2.3.4', dst_port=XX

iptables-scan-multi_ports scenario:
  → Filter: log_type == 'iptables_drop' AND service == 'tcp'
  → Group by: source_ip
  → Count: distinct dst_port
  → Overflow: when distinct ports >= capacity within leakspeed
```

**low parsed count in metrics is NORMAL.** kern.log has mostly ALLOW/AUDIT lines. The parser only matches DROP/BLOCK events. 97 parsed out of 43k lines is expected for a lightly-scanned host.

## Performance Checklist After Changes

```bash
# Full metrics dump
sudo cscli metrics

# Key indicators:
# 1. kern.log "Lines parsed" — should be > 0 (parser is running)
# 2. iptables-logs parser "Hits" — should be > 0 (parser is matching lines)
# 3. iptables-scan-multi_ports "Poured" — should be > 0 (events reaching scenario)
# 4. iptables-scan-multi_ports "Overflows" — ALERTS GENERATED (not `-`)
# 5. Bouncer "active_decisions" — should increase (bans being enforced)
```
